# Contradiction Matrix (Altshuller 39×39)

The matrix is the classic TRIZ lookup: you state an **engineering contradiction**
as two of the 39 standard parameters — one you want to *improve*, one that gets
*worse* — and the matrix returns the inventive principles that historically
resolved that exact trade-off across millions of patents. It is a shortcut into
the 40 principles, not a separate method.

> The matrix narrows 40 principles down to ~4 high-probability ones. It does not
> "solve" the problem — it hands you the most promising principles to apply. You
> still do the inventive work of mapping a principle onto your system.

## How to use it (the loop)

1. **Frame the contradiction in plain language.** "If we increase the wall
   thickness (improve strength), the part gets heavier (weight worsens)."
2. **Map each side to one of the 39 parameters.** Use the table below. Pick the
   *closest* standard parameter; it rarely matches perfectly — that's expected.
   - The thing you are trying to gain → **improving** parameter.
   - The thing that degrades as a side effect → **worsening** parameter.
3. **Look up the cell** (improving row, worsening column) → a short list of
   inventive principle numbers. Run the tool:
   `python .claude/skills/triz-innovation/scripts/triz_matrix.py 14 1`
   (improving = 14 Strength, worsening = 1 Weight of moving object).
4. **Read each suggested principle** in `references/inventive-principles.md` and
   force a concrete interpretation onto *your* system. Reject the ones that don't
   map; keep the 1–2 that spark a real solution.
5. **Tag the solution** with the principle, e.g. `[IP-40 Composite materials]`.

## Important caveats (so you don't misuse it)

- **Empty / diagonal cells.** When improving and worsening are the *same*
  parameter, the matrix says "see physical contradiction" — drop to
  `references/physical-contradictions.md` and separate in time/space/condition.
  Some off-diagonal cells are also empty (no strong historical pattern); when the
  tool returns nothing, fall back to Resource Analysis + IFR.
- **Parameter choice dominates the result.** The same problem framed with
  different parameters yields different principles. If the suggestions feel weak,
  **re-map the parameters** and look again — try 2–3 framings.
- **"Moving" vs "stationary"** matters: a moving object's weight (1) is a
  different parameter from a stationary object's weight (2). Choose by whether the
  element changes position during the operating cycle.
- The matrix is **engineering-flavored** (force, weight, temperature…). For
  business/software/clinical problems, translate: "weight" → overhead/payload,
  "speed" → latency/throughput, "stress" → load/pressure on a team, "reliability"
  → uptime/adherence. The principles still transfer; see the domain references.

## Worked example

> "Our onboarding flow needs more validation steps to cut fraud (improve
> reliability), but each step makes signup slower and users drop off (loss of
> time worsens)."

- Improving = **27 Reliability**; worsening = **25 Loss of time**.
- `python scripts/triz_matrix.py 27 25` → principles like
  **10 Preliminary action**, **30 Flexible shells/films**, **4 Asymmetry**,
  **34 Discarding & recovering** (illustrative — read what the tool returns).
- Interpretation: *10 Preliminary action* → pre-verify identity in the background
  before the user arrives (device fingerprint, pre-filled data) so the visible
  flow stays short. Tagged `[IP-10 Preliminary action]`. Then test per stage 10.

## Where this sits in the pipeline
Use the matrix inside **stage 4 (Contradiction analysis)** once you've named an
engineering contradiction. If the contradiction collapses to a single element
needing opposite states, switch to **physical contradiction + separation**.
